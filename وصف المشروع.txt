مشروع: تطبيق ومنصة إلكترونية متعددة الخدمات للملابس (بيع – تأجير – تبرع – طلب ملابس)

آخر تحديث: 2026-01-24
الجهة: الشركة / فريق العمل

1. الملخص التنفيذي
يهدف المشروع إلى إنشاء منصة رقمية موحدة وآمنة في سوريا، تجمع بين بيع الملابس، تأجيرها، التبرع بها، وطلب ملابس حسب الحاجة. تعالج المنصة مشاكل التشتت، ضعف الثقة، وصعوبة الوصول إلى الملابس المناسبة من خلال نظام مركزي يخضع للإشراف الإداري، ويضمن جودة المحتوى، وحماية الأطراف، وإدارة العمليات (بيع/تأجير/تبرع) من البداية حتى التسليم أو الإرجاع.
المنصة تعمل كوسيط منظم (Marketplace) مع موافقة إدارية إلزامية على كل محتوى يُنشر لضمان الموثوقية والالتزام بالقوانين والسياسات.
________________________________________
2. أهداف المشروع
• الهدف الأساسي
إنشاء منصة موثوقة وشاملة للملابس في سوريا (بيع – تأجير – تبرع – طلب خاص).
• الأهداف التجارية
•	تحقيق إيرادات من:
o	عمولة على عمليات البيع والتأجير
o	اشتراكات شهرية/سنوية للبائعين (باقات مميزة)
o	رسوم خدمات (توصيل – تمييز لباس)
• الأهداف الخدمية
•	تسهيل وصول المستخدمين إلى الملابس المناسبة حسب الحاجة والميزانية
•	تمكين الأفراد من بيع أو تأجير أو التبرع بملابسهم بسهولة
•	خلق فرص عمل في التوصيل والإدارة
• الأهداف المجتمعية
•	دعم ذوي الدخل المحدود
•	تعزيز ثقافة إعادة الاستخدام
•	تشجيع التبرع المنظم بدل العشوائي
________________________________________
3. أنواع المستخدمين (User Roles)
1. السوبر أدمن (Super Admin)
•	مالك المنصة
•	إدارة الصلاحيات الكاملة
•	إضافة/إزالة الأدمنز
•	التحكم بالإعدادات العامة والسياسات
2. الأدمن (Admin)
•	مراجعة والموافقة على:
o	المنتجات المعروضة (بيع/تأجير/تبرع)
o	طلبات الملابس
•	إدارة المحتوى والتصنيفات
•	التعامل مع الشكاوى والنزاعات
3. المستخدم (User)
•	مستخدم عادي يمكنه:
o	تصفح المنصة
o	شراء ملابس
o	استئجار ملابس
o	التبرع بملابس
o	طلب لباس معين
4. مندوب التوصيل (Delivery Agent)
•	استلام وتسليم الطلبات
•	توثيق حالة الملابس
•	تحصيل الدفعات (إن وجدت)
________________________________________
4. أقسام المنصة الرئيسية
1. قسم بيع الملابس
•	عرض ملابس جديدة أو مستعملة للبيع
•	تحديد السعر، الحالة، الصور، المقاسات
•	لا يظهر أي غرض إلا بعد موافقة الأدمن
2. قسم تأجير الملابس
•	تأجير ملابس المناسبات
•	تحديد مدة الإيجار، السعر، العربون
•	إدارة التوفر الزمني
3. قسم التبرع بالملابس
•	تمكين المستخدم من عرض ملابس للتبرع
•	تحديد الفئة (رجالي – نسائي – أطفال)
•	مراجعة المحتوى قبل النشر
4. قسم طلب الملابس (Requests)
•	زر: "اطلب لباسًا"
•	المستخدم يكتب طلبه، مثال:
o	"أحتاج حذاء رجالي لمناسبة عرس، مقاس 42"
•	يظهر الطلب في قسم الطلبات بعد موافقة الأدمن
•	يمكن لمستخدمين آخرين الرد على الطلب عبر:
o	عرض بيع
o	عرض تأجير
o	عرض تبرع

5. سير عمليات المنصة (Flow العمليات)
توضّح الفقرات التالية ترتيب الخطوات وانتقال الحالات لكل عملية على المنصة.

أ. مسار «إضافة غرض» (بيع / تأجير / تبرع)
1. المستخدم يختار «إضافة غرض» ثم نوع العملية (بيع، تأجير، أو تبرع).
2. يملأ البيانات: صور، وصف، مقاسات، سعر (إن وجد)، حالة (جديد/مستعمل)، وعيِّنات إن وُجدت.
3. يُرسل الطلب → الحالة: «مُرسل للمراجعة» (لا يظهر للمتصفّحين).
4. الأدمن يراجع في لوحة التحكم:
   • موافقة → اللباس ينتقل إلى «موافق عليه» ويُعرض في القسم المناسب.
   • رفض → يُعاد إلى «مرفوض» مع إمكانية إبلاغ المستخدم بوجه الرفض (إن وُجد).
5. بعد الموافقة: المستخدمون يرون اللباس، ويمكنهم «التواصل» أو «عرض سعر» حسب آلية المنصة.

ب. مسار «طلب لباس»
1. المستخدم يختار «اطلب لباسًا» ويدخل تفاصيل الطلب (نوع اللباس، المقاس، المناسبة، إلخ).
2. يُرسل الطلب → الحالة: «مُرسل للمراجعة» (لا يظهر في قسم الطلبات).
3. الأدمن يراجع:
   • موافقة → الطلب يظهر في قسم «طلبات الملابس».
   • رفض → يبقى غير منشور مع إبلاغ مقدم الطلب إن أمكن.
4. بعد الموافقة: مستخدمون آخرون يرون الطلب ويمكنهم تقديم عروض (بيع / تأجير / تبرع).

ج. مسار «الرد على طلب لباس» (عرض بيع / تأجير / تبرع)
1. مستخدم يشاهد طلباً معتمداً ويرغب بالاستجابة → يختار «قدّم عرضاً».
2. يحدّد نوع العرض (بيع، تأجير، أو تبرع) ويدخل التفاصيل والسعر إن لزم.
3. يُرسل العرض → يُربط بالطلب ويُبلّغ صاحب الطلب (بحسب آلية الإشعارات).
4. صاحب الطلب يراجع العروض ويتواصل مع مقدم العرض خارج المنصة أو عبر وسائل التوثيق المعتمدة (واتساب، هاتف، إلخ).

د. مسار الموافقة الإدارية (مشترك)
• كل محتوى جديد (لباس أو طلب) يمرّ بحالة «مُرسل للمراجعة».
• لا يُنشر شيء للمتصفّحين إلا بعد انتقاله إلى «موافق عليه».
• الرفض يوقف النشر مع إمكانية توضيح السبب للمستخدم.
• إعادة الإرسال للمراجعة (بعد تعديل المحتوى) تعيد الدورة من الخطوة 3 في المسار المعني.

ه. ملخص تسلسلي مبسط
لباس: [إنشاء] → [مراجعة أدمن] → موافقة → [منشور] | رفض → [غير منشور].
طلب: [إنشاء] → [مراجعة أدمن] → موافقة → [منشور] → [عروض من مستخدمين] → تواصل خارجي/داخلي.

6. المتطلبات الوظيفية التفصيلية
أ. التسجيل والمصادقة
•	إنشاء حساب (بريد إلكتروني – كلمة مرور؛ هاتف اختياري)
•	توثيق البريد الإلكتروني (إعادة تعيين كلمة المرور متاحة)
•	قبول الشروط والسياسات
•	ملاحظة: التوثيق عبر SMS ممكن إضافته لاحقاً حسب المتطلبات
________________________________________
ب. إضافة غرض (بيع / تأجير / تبرع)
•	زر: "إضافة غرض"
•	إدخال:
o	نوع العملية (بيع – تأجير – تبرع)
o	الصور
o	الوصف
o	المقاسات
o	السعر (إن وجد)
•	الحالة الافتراضية: بانتظار موافقة الأدمن
________________________________________
ج. طلب لباس
•	زر: "اطلب لباسًا"
•	إدخال تفاصيل الطلب
•	لا يظهر الطلب إلا بعد موافقة الأدمن
•	استقبال عروض من مستخدمين آخرين

•	Backend: Laravel 12.x
•	Dashboard: Filament 5
•	Frontend: Blade، Tailwind CSS، Alpine.js
•	أدوات البناء: Vite
